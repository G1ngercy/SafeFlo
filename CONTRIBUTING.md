# Contributing to SafeFlow

Спасибо за интерес к вкладу в SafeFlow! Этот документ объясняет, как мы работаем, какие изменения принимаем, и каких принципов придерживаемся.

## Главное правило

> **Безопасность важнее функциональности.** Если у вас отличная фича, но она требует сетевого вызова, install-скрипта или модификации файлов вне проекта — мы её, скорее всего, не примем. Это не потому, что идея плохая, а потому что SafeFlow определяется именно этими ограничениями.

## Какие PR мы охотно принимаем

- **Исправления багов** с тестом, демонстрирующим проблему.
- **Исправления безопасности** (см. ниже про процедуру для уязвимостей).
- **Улучшения существующих модулей** — больше тестов, более понятные сообщения об ошибках, лучшая документация.
- **Дополнительные тесты регрессии**, особенно для security-инвариантов.
- **Перевод документации** на другие языки.
- **Улучшения производительности** при условии, что они не усложняют код.

## Какие PR мы, скорее всего, отклоним

- Добавление новых зависимостей без серьёзного обоснования.
- Добавление сетевых вызовов в основной код.
- Добавление install/postinstall/preinstall скриптов в `package.json`.
- "Enterprise-фичи", которые на самом деле не работают (заглушки, фейковые данные).
- Большие реорганизации кода без обсуждения в issue.
- Изменения, расширяющие поверхность атаки (новые типы агентов с сетевым доступом, federation, и т.п.).

Если вы не уверены, подойдёт ли ваше предложение — **сначала откройте issue**. Это сэкономит время всем.

## Процесс

### Для багов

1. Проверьте, нет ли уже issue по этой проблеме.
2. Откройте issue, заполнив [bug template](.github/ISSUE_TEMPLATE/bug_report.yml).
3. Если хотите сами исправить — упомяните это в issue, мы назначим его на вас.
4. Сделайте fork, ветку `fix/issue-XXX-краткое-описание`, реализуйте, добавьте тест.
5. Откройте PR. CI должен быть зелёным.

### Для фич

1. **Сначала** откройте issue с feature request. Опишите проблему и предложение.
2. Дождитесь обсуждения и согласия мейнтейнера. Это нужно, чтобы вы не тратили время на работу, которую не примут.
3. Сделайте fork, ветку `feat/название`, реализуйте, добавьте тесты и документацию.
4. Откройте PR.

### Для документации

Документационные PR без предварительного issue — добро пожаловать.

## Уязвимости безопасности

**НЕ ОТКРЫВАЙТЕ публичный issue для уязвимостей.**

Используйте приватный security advisory: на странице репозитория → Security → Report a vulnerability.

Подробности — в [SECURITY.md](SECURITY.md). Мы стремимся ответить в течение 3 рабочих дней.

В отличие от некоторых проектов, мы **не** игнорируем сторонние security-PR. Если вы прислали фикс уязвимости, мы:

- Ответим в течение недели.
- Объясним, что мы планируем сделать (принять, попросить изменения, сделать своё исправление).
- Если используем вашу работу — добавим credit в release notes.

## Среда разработки

```bash
git clone https://github.com/G1ngercy/SafeFlo.git
cd safeflow

# Используем npm ci, а не npm install — он строго следует package-lock.json
npm ci

npm run build    # компиляция TypeScript
npm test         # запуск тестов
npx tsc --noEmit # проверка типов
```

### Node.js версии

Поддерживаем `Node.js >= 20.0.0`. Если ваше изменение требует более новой версии — обсудите это в issue.

## Стиль кода

- TypeScript в strict mode. Никаких `any` без явного обоснования в комментарии.
- Все публичные функции должны иметь JSDoc.
- Тесты обязательны для нового кода.
- Имена коммитов — на любом языке, но желательно с префиксом: `fix:`, `feat:`, `docs:`, `refactor:`, `test:`, `security:`.

## Чек-лист перед PR

Полный чек-лист — в [.github/pull_request_template.md](.github/pull_request_template.md), но кратко:

- [ ] Нет install-скриптов в `package.json`.
- [ ] Нет новых сетевых вызовов в основном коде.
- [ ] Нет `eval`/`Function(...)`/`exec()` на пользовательских данных.
- [ ] Все SQL-запросы — параметризованные.
- [ ] Все пути — через `safeResolve`.
- [ ] Все входные данные — через Zod.
- [ ] Если меняли MCP descriptions — `node scripts/audit-mcp-descriptions.mjs` проходит.
- [ ] `npm test`, `npm run build`, `npx tsc --noEmit` все зелёные.
- [ ] Добавлены тесты для изменений.
